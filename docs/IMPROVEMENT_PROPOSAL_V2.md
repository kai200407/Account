# 智能进销存系统改进方案 v2.0（终版）

> 经 CodeBuddy (5/10) + Codex (4/10) 两轮对抗评审后综合输出
> 
> 定位：7人小团队、日活100-500、Next.js + Prisma + SQLite/PostgreSQL

---

## 对抗评审共识总结

### 双方一致同意删除的项目
| 项目 | 删除理由 |
|------|----------|
| CSRF 防护 | Bearer Token 架构天然防 CSRF，画蛇添足 |
| 暗黑模式 | 7人记账系统，优先级严重错误 |
| PWA 完整支持 | 复杂度远超收益，先做好移动端表单体验 |
| 手势操作 | 高价值财务数据不适合滑动删除，易误触 |
| SWR/React Query（全局） | 交易链路不适合客户端缓存，先做查询收敛 |
| Sentry | 没有错误边界和测试时，监控只收噪音 |
| API 性能监控平台 | 日活 500 不需要专门平台 |
| CMD+K 全局搜索 | 7人团队不会形成肌肉记忆，过度设计 |

### 双方一致同意保留的项目
- 成本核算测试（提升到 P0）
- 金额精度修复（修泄漏点，不是全链路重写）
- API 输入校验（zod，金额类 API 优先）
- FEAT-17 订单修改（Backlog P0 刚需）
- 登录限流
- Error Boundary
- 开单草稿自动保存
- 查询收敛优化

### 核心分歧及裁决

| 分歧点 | CodeBuddy | Codex | 裁决 |
|--------|-----------|-------|------|
| 登录限流方案 | 内存 Map 够用 | 数据库级更稳 | **内存 Map 先行**（7人单实例），迁移 PG 后改数据库级 |
| Middleware 鉴权 | 要做，纵深防御 | 不做，用 withAuth | **后置到 P2**，先补 withAuth 包装器 + ESLint 约束 |
| 多租户隔离 | 审计扫描所有 API | 补隔离回归测试 | **补测试为主**，不搞专门审计工程 |
| 数据备份 | 方案严重不足 | 已有备份，补恢复演练 | **Codex 对**，备份已存在，补恢复验证即可 |
| 底部导航栏 | 要做 | 已做了 | **Codex 对**，sidebar.tsx 已有手机底部导航 |
| 前端缓存 | 整体不做 | 只读页面可做 | **局部可选**，Dashboard/报表后续可加，交易页不加 |

## 最终 Top 10 优先级

### P0 — 必须立即做（影响数据正确性）

#### 1. 核心测试：成本核算单测 + 事务集成测试
- **为什么**: 零测试 + 金额计算 = 定时炸弹
- **范围**: 
  - `cost-calculation.ts` 4个函数的单元测试（边界：库存为0、负数、超大金额）
  - 进货→成本更新→销售→利润计算 的端到端集成测试
  - 退货→成本退回→库存恢复 的回滚测试
- **工时**: 4h

#### 2. 金额精度修复
- **为什么**: `cost-calculation.ts` 用 JS `number` 计算，`round2` 散落各处
- **做什么**: 
  - 引入 `decimal.js`，修复 `cost-calculation.ts` 的 number 泄漏
  - 统一舍入规则：所有金额 `toFixed(2)` 改为 Decimal 精确舍入
  - 不需要改全链路，只修 TypeScript 计算层
- **工时**: 4h

#### 3. FEAT-17 订单修改功能
- **为什么**: Backlog 唯一 P0，最易出错的业务链路
- **做什么**:
  - 取消旧单 → 库存回滚 + 余额调整 + 利润冲销
  - 重建新单 → 重新计算成本/库存/余额
  - 全程事务保证原子性
- **工时**: 8h

#### 4. 并发库存链路补强
- **为什么**: 虽然 `createStockMovement` 已用 `increment` + 条件保护，但跨商品循环更新存在竞态窗口
- **做什么**:
  - 审计所有库存写入路径，确保全在事务内用原子 `increment/decrement`
  - 补并发集成测试：双请求同时销售同一商品，验证不超卖
  - 不加 version 乐观锁（当前规模不需要）
- **工时**: 4h

### P1 — 尽快做（影响安全性和体验）

#### 5. API 输入校验（zod）
- **范围**: `auth/login`, `purchases POST`, `sales POST`, `payments POST`
- **重点**: 金额非负、items 非空、枚举合法、日期格式、字符串长度上限
- **注意**: 和租户归属校验分开，zod 只管格式
- **工时**: 4h

#### 6. 多租户隔离回归测试
- **做什么**: 关键 API 补 tenantId 过滤测试、资源归属测试
- **防漏**: 添加 `withAuth(handler)` 包装器，减少新接口遗漏风险
- **工时**: 3h

#### 7. 登录限流 + 失败审计日志
- **方案**: 内存 Map 限流（5次/分钟 → 锁15分钟）+ 失败审计日志
- **迁移 PG 后**: 改数据库级计数
- **工时**: 3h

#### 8. Error Boundary + 加载/提交反馈
- **做什么**: 
  - `error.tsx` + `global-error.tsx` 防白屏
  - 关键页面 `loading.tsx`（Next.js 内置）
  - 提交按钮 loading 状态
  - Dashboard 骨架屏
- **工时**: 3h

### P2 — 稳步推进（代码质量和体验）

#### 9. 开单草稿自动保存
- **做什么**: 新建订单表单数据存 `sessionStorage`，带过期时间+版本号
- **注意**: 不自动恢复已提交数据，避免重复提交
- **工时**: 2h

#### 10. 查询收敛优化
- **做什么**:
  - 列表 API `select` 必要字段，详情页再 `include`
  - 库存预警改数据库筛选（当前全量加载再内存过滤）
  - Dashboard 精简投影
- **工时**: 3h
