# 智能进销存系统改进方案 v1.0

> 作者: Claude | 状态: 待对抗评审

## 一、安全加固（P0 紧急）

### 1.1 登录防暴力破解
- **现状**: 登录 API 无任何限流，攻击者可无限尝试密码
- **方案**: 添加内存级 rate limiter（基于 IP + 手机号，5次/分钟失败后锁定15分钟）
- **实现**: 使用 Map 存储尝试次数，无需引入 Redis

### 1.2 Next.js Middleware 鉴权
- **现状**: 每个 API 手动调用 requireAuth，容易遗漏
- **方案**: 创建 src/middleware.ts 统一拦截 /api/* 路由（排除 /api/auth/*）
- **收益**: 防止新 API 忘记加鉴权

### 1.3 输入校验
- **现状**: API 仅检查字段是否存在，无类型/格式/范围校验
- **方案**: 引入 zod 做 schema 级校验（手机号格式、金额范围、字符串长度）
- **优先级**: 先覆盖 auth + purchases + sales 三个高危路由

### 1.4 CSRF 防护
- **现状**: 纯 JWT Bearer Token，无 CSRF 保护
- **方案**: API 已用 Authorization Header 而非 Cookie，CSRF 风险低，但建议添加 SameSite Cookie 属性作为二层防护

### 1.5 密码强度
- **现状**: 注册时无密码强度要求
- **方案**: 最少6位，包含数字和字母

## 二、性能优化（P1 重要）

### 2.1 API 查询优化
- **现状**: 部分 API 使用 include 加载大量关联数据
- **方案**: 
  - 列表页只 select 必要字段
  - Dashboard API 使用 Promise.all 并行查询（部分已做）
  - 库存预警使用数据库筛选替代内存过滤（当前是全量加载再过滤）

### 2.2 Prisma 连接池
- **现状**: 使用默认连接池配置
- **方案**: SQLite 开发无需调整，但生产 PostgreSQL 需配置 connection_limit

### 2.3 前端缓存
- **现状**: 每次页面切换重新请求数据
- **方案**: 引入 SWR 或 React Query 做客户端缓存 + stale-while-revalidate

### 2.4 图片上传优化
- **现状**: 有上传 API 但无压缩/CDN
- **方案**: 添加图片压缩（sharp）+ 考虑对象存储

## 三、用户体验（P1 重要）

### 3.1 全局搜索
- **现状**: 无全局搜索功能
- **方案**: Command+K 搜索面板，搜索商品/客户/供应商/单据

### 3.2 Error Boundary
- **现状**: 页面 JS 报错直接白屏
- **方案**: 添加 React Error Boundary + 友好错误页面

### 3.3 Loading 骨架屏
- **现状**: 数据加载时无反馈或仅显示文字
- **方案**: 关键页面（Dashboard、商品列表、报表）添加 Skeleton 加载态

### 3.4 表单自动保存
- **现状**: 开单页面刷新数据丢失
- **方案**: 开单表单数据存 sessionStorage，防止意外丢失

### 3.5 暗黑模式
- **现状**: 已安装 next-themes 但未实际使用
- **方案**: 激活暗黑模式切换，所有页面适配 dark: 样式

## 四、数据完整性（P0 紧急）

### 4.1 并发库存操作
- **现状**: 高并发下可能出现库存超卖（读-改-写不是原子的）
- **方案**: Prisma 事务内使用乐观锁（version 字段）或数据库级 increment/decrement
- **备注**: 当前部分操作已用 increment，但不够完整

### 4.2 金额精度
- **现状**: cost-calculation.ts 用 JavaScript Number 计算金额，有浮点精度问题
- **方案**: 使用 Prisma Decimal 类型 + decimal.js 库做精确计算
- **影响**: 成本核算链路全部需要改用 Decimal

### 4.3 数据备份
- **现状**: SQLite 无自动备份
- **方案**: 添加定时备份脚本（cron 每天导出 .db 文件）

## 五、移动端体验（P1 重要）

### 5.1 PWA 支持
- **现状**: 无 manifest.json、无 Service Worker
- **方案**: 添加 PWA 支持，实现「添加到主屏幕」
- **收益**: 手机端体验接近原生 App

### 5.2 底部导航栏
- **现状**: 移动端使用顶部 dropdown 菜单，操作不便
- **方案**: 手机端添加固定底部导航栏（首页/销售/进货/库存/更多）

### 5.3 手势操作
- **现状**: 无滑动手势
- **方案**: 列表页支持左滑删除/编辑，下拉刷新

## 六、代码质量（P2 长期）

### 6.1 测试
- **现状**: 零测试覆盖
- **方案**: 
  - 优先级1: 成本核算函数单元测试（纯函数，最容易测）
  - 优先级2: 关键 API 集成测试（登录、进货、销售）
  - 优先级3: 页面 E2E 测试（Playwright）

### 6.2 API 响应类型统一
- **现状**: API 返回结构不完全统一
- **方案**: 定义统一的 ApiResponse<T> 泛型，所有 API 遵循 { success, data, error, pagination }

### 6.3 代码复用
- **现状**: 多个报表页面有大量重复的表格/卡片/导出逻辑
- **方案**: 抽取 ReportLayout、DataTable、ExportButton 通用组件

## 七、运维监控（P2 长期）

### 7.1 错误监控
- **方案**: 集成 Sentry 捕获前端+后端异常

### 7.2 API 性能监控
- **方案**: 添加请求耗时日志，慢查询告警

### 7.3 健康检查
- **方案**: 添加 /api/health 端点，检查数据库连接
