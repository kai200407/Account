## Codex 评审意见
### 总评分: 4/10
### 一句话: 这不是一份聚焦“小团队、小流量、强业务正确性”系统的改进方案，而是一份把真实问题、已完成事项和想当然优化混在一起的愿望清单。
### 逐项评审:

#### 一、安全加固（P0 紧急）

**1.1 登录防暴力破解: 部分同意**

同意“要做”，不同意“内存级 Map 锁 15 分钟”被包装成可靠方案。这个系统虽然只有 7 人团队，但生产是多进程部署的概率不低，内存限流在重启、扩容、反向代理后都不稳定，最多算“临时止血”。

替代方案：
- 短期先做数据库级失败计数和冷却时间，按 `phone` 为主、`IP` 为辅，避免实例间失效。
- 同时补登录失败审计日志，便于排查恶意尝试和误伤。
- 如果后续真有公网暴露和攻击压力，再上 Redis 或网关限流；现在没必要一步到位。

**1.2 Next.js Middleware 鉴权: 不同意**

这是典型“以为更统一，实际更脆”。当前项目架构已经明确要求 API 显式调用 `requireAuth(request)`，而且 `requireAuth` 不只是验 JWT，还会回源数据库校验用户是否仍有效。把鉴权平移到 `middleware.ts`，并不能替代路由层的租户、角色、资源归属检查，最后大概率变成“双份鉴权逻辑”。

更糟的是，方案把它列为高优先级收益项，但真正的问题并不是“有没有统一入口”，而是“有没有静态约束和测试防漏”。

替代方案：
- 保留路由内显式 `requireAuth`。
- 增加 ESLint 规则、约定测试或一个 `withAuth(handler)` 包装器，减少遗漏风险。
- 对 owner 权限、tenantId 过滤、资源归属检查补测试，比中间件值钱得多。

**1.3 输入校验: 同意**

这是方案里最扎实的 P0/P1 项之一。当前很多路由仍以 `if (!field)` 级别校验为主，复杂业务接口靠手写判断，长期一定会漏边界。

但方案还是太泛。它说“先覆盖 auth + purchases + sales”，这个方向对，但没有把“金额字段、数组项、ID 归属、枚举值、日期格式、分页参数”拆开定义，导致落地时容易只补表面 schema。

替代方案：
- 先覆盖 `auth/login`、`purchases POST`、`sales POST`、`payments POST`。
- 重点校验数量/金额非负、`items` 非空、枚举值合法、日期格式合法、字符串长度上限。
- 把校验层和业务归属校验分开，不要误以为 zod 能解决租户安全。

**1.4 CSRF 防护: 不同意**

这条基本是在凑数。当前系统走的是 `Authorization: Bearer <token>`，不是 Cookie Session，CSRF 不是主风险。方案自己也承认“风险低”，却还塞进 P0 安全加固章节，属于优先级污染。

替代方案：
- 删除这一项。
- 真正该补的是上传接口的文件名策略、MIME/内容校验、以及登录和关键写接口的审计。

**1.5 密码强度: 部分同意**

同意改，但“最少 6 位且字母数字混合”很像完成 KPI，不像安全方案。对这个 7 人内部系统，密码策略不能太重，否则只会诱导口令复用和记在纸上。

替代方案：
- 最低长度提到 8 位即可，不强推复杂组合。
- 配合首次弱密码提醒、owner 重置密码能力、登录失败限流，整体收益更高。

#### 二、性能优化（P1 重要）

**2.1 API 查询优化: 同意**

这是合理项，但优先级要往前挪，因为它是低风险高收益。当前销售/进货列表确实有 `include` 整包拉关联数据的倾向，库存预警全量拉取再过滤如果属实，也属于直接可修的浪费。

替代方案：
- 先改最重的列表 API：列表页 `select` 必要字段，详情页再 `include`。
- 为报表和 Dashboard 建最小投影，而不是泛化为“Promise.all + 并行”口号。
- 先做 SQL/Prisma 查询收敛，再谈前端缓存。

**2.2 Prisma 连接池: 不同意**

这是脱离当前阶段的“云原生姿势正确”。开发环境 SQLite 根本无关，生产 PostgreSQL 的连接池参数也不是当前主要瓶颈。没有任何监控数据支持它进入优先级列表。

替代方案：
- 从方案中降级为部署注意事项。
- 等有真实慢查询、连接打满、p95 抖动数据后再调。

**2.3 前端缓存: 部分同意**

有价值，但不是现在。这个系统是后台管理型进销存，不是高频公共内容站。你上 SWR/React Query 后会立刻引入缓存失效、写后同步、标签页一致性的问题，复杂度明显高于收益。

替代方案：
- 优先使用 Next/React 当前能力和手工局部刷新。
- 先给少数只读页面做缓存，如 Dashboard、报表筛选结果。
- 交易页、库存页、收付款页不要急着上全局客户端缓存，避免显示旧数据。

**2.4 图片上传优化: 部分同意**

方案方向没错，但“压缩 + CDN/对象存储”对当前规模偏重。更现实的问题是：上传能力已经存在，先补安全和存储边界，再谈 CDN。

替代方案：
- 先做服务端尺寸/体积校验、文件名去碰撞、图片元数据处理。
- 如果部署磁盘压力或访问量上来，再迁对象存储。
- `sharp` 可以有，但属于商品图片功能完善，不是系统级性能大项。

#### 三、用户体验（P1 重要）

**3.1 全局搜索: 部分同意**

这是典型“看起来高级，实际不一定先要”。对 100-500 DAU 的小团队系统，全局搜索不是刚需，尤其是当现有模块内搜索和常用项排序还没做到极致时。

替代方案：
- 先把商品/客户/供应商/单据各自列表的搜索、筛选、最近使用、快速跳转做好。
- 如果真做全局搜索，也先支持商品和客户，不要一口气拉全实体 Command+K。

**3.2 Error Boundary: 同意**

投入小、收益稳定，适合尽快做。后台系统白屏会直接打断业务流程，这条比暗黑模式、手势操作都实际。

替代方案：
- 在 App Router 下用 `error.tsx`、`global-error.tsx` 和关键区域降级 UI。
- 记录错误上下文，和后续监控接起来。

**3.3 Loading 骨架屏: 部分同意**

可以做，但别把它当“重要改进”。对多数内部系统页面，明确的 loading 文案、按钮禁用态、局部 skeleton 就够了，没必要全页铺骨架。

替代方案：
- 优先补提交中、刷新中、分页切换中的明确反馈。
- Dashboard 和长表格页可加 skeleton，其他地方克制处理。

**3.4 表单自动保存: 同意**

对开单页是实用改进，且成本不高。这里比 PWA 更贴近真实损失场景。

替代方案：
- 只对“新建订单草稿”做 `sessionStorage`，带过期时间和版本号。
- 不要自动恢复已提交成功的数据，避免误重复提交。

**3.5 暗黑模式: 不同意**

这是最明显的优先级错误之一。`next-themes` 已安装，不代表“应该激活”。对这种白天高频操作的进销存系统，暗黑模式不是核心需求，而且全量适配成本被明显低估。

替代方案：
- 从方案中移除。
- 先修对比度、触控面积、表格可读性、移动端操作路径，这些才是真正 UX。

#### 四、数据完整性（P0 紧急）

**4.1 并发库存操作: 部分同意**

问题存在，但方案给出的解法不聚焦。当前库存写入链路已经在 `createStockMovement` 里使用条件 `updateMany + increment`，这比“普通读改写”安全得多。真正的风险在于订单接口存在事务外预检、库存金额和库存数量更新分步进行、以及跨商品循环更新时的竞态窗口，而不是一句“加 version 乐观锁”就能解决。

替代方案：
- 保持数据库原子 `increment/decrement` 思路，不要轻易全表加 `version`。
- 把“库存是否足够”的最终校验留在事务内，以数据库条件更新为准。
- 补并发集成测试：双请求同时销售同一商品，验证不会超卖且库存金额不漂移。

**4.2 金额精度: 部分同意**

方案指出的问题真实，但“全链路改 Decimal + decimal.js，6 小时搞定”明显不诚实。当前 schema 已经大量使用 Prisma `Decimal`，真正的问题是 `cost-calculation.ts` 和订单金额计算把值转成 `number` 处理，再写回 Decimal 字段。也就是说，不是“从零切换 Decimal”，而是“修复 Number 泄漏点和舍入策略”。

替代方案：
- 先锁定金额/成本计算核心函数，统一在边界做 Decimal 转换。
- 定义统一舍入规则，别让 `round2` 散落在各处。
- 先补单元测试覆盖采购、销售、退货、盘点四条链路，再逐步收口实现。

**4.3 数据备份: 不同意**

这条直接暴露方案作者没核对现状。项目进度文档里已经完成 `scripts/backup.sh` 和部署脚本中的定时备份。把已完成事项重新提成改进项，是严重扣分项。

替代方案：
- 把“有没有备份”改成“有没有恢复演练、保留策略、异地备份校验”。
- 对 SQLite 开发环境可以忽略自动备份，重点看生产 PostgreSQL 的恢复可用性。

#### 五、移动端体验（P1 重要）

**5.1 PWA 支持: 不同意**

这是整份方案里最不务实的高优先级项之一。PWA 会带来缓存、离线、资源更新、安装体验、兼容性的一整套复杂度，而这套系统的核心不是“像原生 App”，而是“别出错、别卡、手机能顺手开单”。

替代方案：
- 先把移动端表单、底部操作区、键盘遮挡、列表密度、扫描录入做好。
- 只有在用户明确提出“经常从桌面图标进入、弱网也要用”时，再评估轻量 manifest。

**5.2 底部导航栏: 不同意**

这条再次说明方案和现状脱节。项目里已经有 `src/components/sidebar.tsx` 的手机底部导航，且 `PROGRESS.md` 明确记录“手机底部导航 5 项”已完成。把已存在能力当成待建设，说明方案缺少最基本的代码核查。

替代方案：
- 如果真要改，应该评审“当前 5 项是否是最常用路径、是否需要重排、是否和顶部更多菜单冲突”，而不是从零提建设。

**5.3 手势操作: 不同意**

左滑删除/编辑、下拉刷新非常容易制造误操作，也不符合后台业务系统的一致交互预期。对于单据、库存、客户这类高价值数据，显式按钮比手势更安全。

替代方案：
- 用更大、更固定的位置按钮。
- 对危险操作增加确认和权限控制。
- 如果要提速，优先做扫码、常用项、快捷筛选。

#### 六、代码质量（P2 长期）

**6.1 测试: 同意**

这里反而被低估了。对这类系统，测试不是“长期可选”，而是保障库存和金额正确性的必要投资。特别是当前已经有成本核算、退货、盘点、调拨、仓库库存等复杂链路，靠手工验证迟早出事。

替代方案：
- 把“成本核算单元测试 + 销售/进货/收付款集成测试”抬到 Top 3。
- E2E 可以后放，但 API/事务测试不能继续拖。

**6.2 API 响应类型统一: 部分同意**

有必要，但不应泛化成一个“大一统 `ApiResponse<T>` 重构项目”。当前更实用的是约束分页响应和错误响应形态，而不是把所有返回结构一次性标准化。

替代方案：
- 先统一列表响应 `{ success, data/items, total, page, totalPages }`。
- 再统一错误响应和前端解析逻辑。
- 类型重构跟随新增/改动接口渐进推进。

**6.3 代码复用: 部分同意**

方向成立，但要小心“为了 DRY 而 DRY”。报表页如果业务差异大，强抽 `ReportLayout/DataTable/ExportButton` 可能只会制造更多 props 和条件分支。

替代方案：
- 先抽真正重复、稳定的部分：日期筛选条、导出触发器、空态/加载态。
- 表格列定义和业务汇总卡片保持按页面独立。

#### 七、运维监控（P2 长期）

**7.1 错误监控: 部分同意**

能提升排障效率，但对这个体量不是第一批。没有统一错误边界、没有关键路径测试时，先上 Sentry 很容易变成“噪音收集器”。

替代方案：
- 先把关键接口 `console.error` 和审计日志结构化。
- 再接轻量错误监控，限定只抓生产异常和关键页面崩溃。

**7.2 API 性能监控: 部分同意**

应做，但不需要“慢查询告警平台化”那一套。现阶段做基础耗时日志和几条关键报表接口的采样就够了。

替代方案：
- 对 `dashboard/reports/sales/purchases` 记录耗时和返回记录数。
- 先定位最慢 3 个接口，再决定是否需要更重的监控。

**7.3 健康检查: 部分同意**

可以有，但别自我感动。`/api/health` 只证明“现在能连上数据库”，不证明业务链路健康。它适合作为部署和容器探活，不适合拿来当核心改进项。

替代方案：
- 如果要加，就做最小版本：进程存活 + DB ping。
- 不要把它排进 Top 10。

### 最大盲点:

这份方案最大的盲点不是某一个技术点，而是完全没有围绕“业务正确性回归”组织优先级。这个系统现在最容易出事故的地方，不是没有 PWA、没有暗黑模式、没有 Middleware，而是订单修改/取消、库存金额一致性、跨仓库库存、收付款余额、退货回滚这些链路在持续迭代后是否还能保持正确。

更具体地说，方案遗漏了四件真正该前置的事：
- `FEAT-17 订单修改功能` 仍是 backlog 的 P0，却没有进入方案。
- 库存/金额/余额/利润的回归测试体系没有被抬到最高优先级。
- 现有“已完成能力”和“待改进能力”没有做事实核验，导致提案可信度下降。
- 没有区分 SQLite 开发环境和 PostgreSQL 生产环境在并发、事务、部署上的差异验证。

### 我的 Top 10 优先级:

1. API 输入校验落地到 `auth/login`、`purchases POST`、`sales POST`、`payments POST`
2. 成本核算与金额精度修复，先消灭 `number` 计算泄漏，再统一舍入规则
3. 补核心测试：成本核算单测 + 销售/进货/收付款事务集成测试
4. 修正并发库存链路，把最终库存校验收敛到事务内数据库原子更新
5. 完成 `FEAT-17 订单修改功能`，并验证库存/余额/利润回滚与重建
6. 查询收敛：重列表接口 `select` 化、库存预警改数据库筛选、Dashboard 精简投影
7. 登录限流与失败审计，先用数据库级冷却策略，不急着上 Redis
8. Error Boundary + 关键页面 loading/提交反馈
9. 开单草稿自动保存（仅新建单据，带过期时间）
10. 上传能力补安全边界与压缩，而不是优先做 PWA、暗黑模式、手势交互
