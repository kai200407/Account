## Codex 第二轮反驳

### 与 CodeBuddy 的核心分歧:

1. 多租户隔离当然重要，但“多租户隔离审计是最大盲点”这个表述不够准。
真正该优先补的是“多租户隔离回归验证”和“防漏机制”，不是先再造一层审计概念。项目架构已经写明所有查询必须带 `tenantId`，测试清单也明确要求验证多租户隔离；当前更高价值的动作是给关键 API 补隔离测试、加静态约束或包装器，防止新接口漏写过滤，而不是把问题抽象成新的审计工程。

2. “数据备份方案严重不足”只能部分成立，前提必须先承认备份已经存在。
`docs/PROGRESS.md` 和 `scripts/backup.sh` 已明确落地“每日自动备份，保留 30 天”，所以不能把它继续描述成“没有备份方案”。如果要批评，正确批评点应该是：缺少恢复演练、异地副本校验、恢复时长目标和定期抽检。问题是“恢复可信度不足”，不是“备份为空白”。

3. 我不同意把 SWR/React Query 直接定性为“过度设计到不该考虑”。
对销售、进货、收付款这些强一致写路径，我同意现在不该上全局客户端缓存；但对 Dashboard、报表这类读多写少页面，局部缓存并非原罪。更准确的结论应该是“不能一刀切全站引入”，而不是“这个方向整体错误”。判断边界要按页面特性，而不是按技术名词。

4. CodeBuddy 说 FEAT-17 是业务刚需，这点对；但如果他的结论是“Codex 忽略了它”，那就是误读。
我在第一轮已经把 `FEAT-17 订单修改功能` 明确列进最大盲点和 Top 10，且依据来自 `docs/BACKLOG.md` 的 P0 定义。真正被忽略的是原始改进方案，不是我的评审。

5. 我同意 CMD+K 全局搜索优先级过高，但不同意把“搜索能力”整体降成可有可无。
更合理的切法不是做一个全实体 Command Palette，而是先把商品、客户、供应商、单据列表内搜索、筛选、最近使用做到位。问题在于方案切得太大，不在于“搜索效率优化”本身不值得做。

### CodeBuddy 说对了什么:

- 成本核算相关测试不该放在 P2。对库存、金额、利润会联动的系统，成本核算单测至少是 P0/P1 交界，实际更接近 P0。
- 暗黑模式是明显的优先级错误，应该从当前改进列表里移除。
- CSRF 在 Bearer Token 架构下不是当前主风险，删除是对的。
- `FEAT-17 订单修改功能` 确实是业务刚需，而且比很多体验型优化都更前。
- CMD+K 全局搜索如果做成全局实体检索，确实是过度设计。

### CodeBuddy 哪里说错了:

- 把“多租户隔离”直接等同于“多租户隔离审计”，容易把问题带偏。现阶段最大风险不是缺少审计口号，而是缺少隔离回归测试、资源归属校验和新接口防漏机制。
- 把备份描述成“严重不足”可以，但前提要先承认已有自动备份。否则就和原方案把已完成事项当待做事项犯了同样的事实错误。
- 把 SWR/React Query 一概打成过度设计太绝对。它不适合先进入交易链路，但在只读报表/Dashboard 上仍可作为后续局部优化选项。
- 如果 CodeBuddy 的表述带有“Codex 忽略 FEAT-17”的意思，这个判断不成立。我的第一轮已经把它提升到核心缺口和前五优先级。

### 最终修正版 Top 10:

1. `FEAT-17` 订单修改功能：按“取消旧单 + 重建新单”实现，并验证库存、余额、利润完整回滚与重算
2. 成本核算与金额精度修复：清理 `number` 泄漏点，统一 Decimal 边界和舍入规则
3. 核心测试前置：先补成本核算单测，再补销售/进货/收付款/订单修改的事务集成测试
4. 多租户隔离回归验证：关键 API 补 tenantId 过滤测试、资源归属测试，并减少新接口漏防护概率
5. 并发库存链路修正：把最终库存校验收敛到事务内原子更新，验证不会超卖或造成库存金额漂移
6. API 输入校验：优先覆盖 `auth/login`、`purchases POST`、`sales POST`、`payments POST`
7. 登录限流与失败审计：采用数据库级失败计数/冷却，不急着引入 Redis
8. 查询收敛与只读页性能优化：先做列表 `select` 化、库存预警数据库筛选、Dashboard/报表精简投影
9. Error Boundary + 提交/加载反馈：先保证白屏降级、提交态、分页切换反馈，而不是铺满骨架屏
10. 备份方案补完到“可恢复”：在现有自动备份基础上增加恢复演练、保留策略校验和异地副本检查
